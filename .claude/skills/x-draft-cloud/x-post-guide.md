# X投稿 トーンガイド

## スタイル

- 冷静でカジュアル。エンジニアの普段の口調
- 「〜した」「〜だった」「〜にした」程度のテンション
- 過度な絵文字、「！！！」、テンション高めの表現は使わない
- 内容が少ないなら短く。水増ししない
- 技術的な内容はユーザー視点に変換する（PR番号やissue番号はそのまま出さない）

## 良い例

- 「peppercheckに通知機能を追加した。設定画面からON/OFFを選べる」
- 「認証周りのバグを修正。ログイン後にセッションが切れる問題が直った」
- 「今日はUI周りの改善をしていた。地味だけど使いやすくなったと思う」
- 「新しい画面の設計を考えていた。もう少し煮詰める」

## 悪い例

- 「🎉✨ 新機能リリース！！通知機能が使えます！ぜひ使ってみてください！！🚀」（テンション高すぎ、絵文字過多）
- 「本日はPR #42をマージし、issueを3件クローズしました」（技術的すぎ）
- 「今日も一日開発頑張りました！明日も頑張ります！」（内容がない）
- 「弊サービスのpeppercheckをぜひご利用ください！」（広告臭い）

## NGパターン

- 絵文字の多用
- 「！」の連続使用
- 「ぜひ」「みなさん」などの呼びかけ
- PRやissue番号をそのまま出す
- 「リリースしました」だけで具体的な内容がない
- ハッシュタグの乱用
- 短い内容を長く引き伸ばす

## 投稿タイプ

素材に応じて以下の投稿タイプを使い分ける。

| タイプ | フォーカス | 例 |
|---|---|---|
| progress | 何をしたか + ユーザーにとって何が変わるか | 「peppercheckに通知機能を追加した。設定画面からON/OFFを選べる」 |
| technical | 何を選んだか + なぜそれにしたか | 「状態管理にRiverpodを選んだ。Provider多すぎ問題がなくて見通しがいい」 |
| problem-solution | 何にハマったか + どう解決したか | 「Neonの接続プールでハマった。pgbouncerのprepared statement設定が原因だった」 |
| reflection | 気づき・考察（短く、余韻を残す） | 「OSSでやると設計の意図をドキュメントに残す習慣がつく」 |

## アルゴリズム対策ルール

- リンクはメイン投稿に絶対に含めない。リンクはセルフリプライに回す
- ハッシュタグは最大2個。具体的な技術名（例: `#Flutter`, `#Riverpod`）を優先。reflectionタイプの場合は `#個人開発` や `#OSS` を使う
- privateリポジトリの内部実装詳細（DBスキーマ、API設計、内部アーキテクチャ）は投稿しない。判断理由と設計思想にフォーカスする
- design docsから投稿する場合は技術選定の「なぜ」にフォーカスし、実装の「どう」は省く
- 画像がある場合（PRスクリーンショット、UI画像）は必ず添付する
